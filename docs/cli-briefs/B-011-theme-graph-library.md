# B-011: Theme + Sub-Scene Graph Library

> Brainstormed with Vanessa 2026-04-26 after watching Kellen play. Core insight: kid input should drive output more directly. B-010's matcher made backgrounds match free-text setting; B-011 goes further by introducing a structured theme + sub-scene graph so kids navigate coherent worlds (real castle geography, real underwater flow), not random rooms. Replaces the matcher as the primary path; matcher stays only as silent fallback for pre-B-011 worlds.

## Goal

Ship a 6-theme × 15-sub-scene background library with branching graph traversal, plus theme-first landing flow with prompt preview reveal. Kid picks a theme, picks (or describes) a starting sub-scene, picks a character, types goal + twist, sees the assembled prompt, then summons. Stories generated by Claude follow library geography for the first 3-4 scenes, then open up for narrative branching.

## Decisions Locked

(from the brainstorm — all 6 Q&A turns + design walk-through approved by Vanessa)

- **6 themes total:** Castle and Dragons, Forest, Candy Land, City, Space, Underwater. List is extensible later — just add a catalog entry.
- **15 sub-scenes per theme = 90 backgrounds total.** Ship as placeholder SVGs in B-011 (CLI generates inline via Claude during implementation). Real art swaps in later batches without touching the catalog.
- **Branching graph per theme.** Each sub-scene is a node with `connects_to: []` edges defining geographic adjacency. Multiple entry-candidate nodes; multiple ending-candidate nodes.
- **Hybrid traversal: graph-anchored entry + story-driven divergence.** Scene 1 = the kid's picked entry sub-scene. Scenes 2-4 must follow library `connects_to` edges from the previous scene (real navigation feel). Scenes 5+ can use any sub-scene from the same theme (Claude bridges narratively, "a hidden door appears..."). Endings must use sub-scenes flagged `can_be_ending: true`.
- **Sub-scene metadata: just-enough.** `{ id, theme, label, description, file_path, connects_to: [], can_be_entry: bool, can_be_ending: bool }`. No mood/depth tags — Claude infers from labels.
- **Setting input: picker grid + free-text fallback.** After theme is locked, kid sees 15-card sub-scene grid filtered to that theme. Below the grid: optional "or describe a different starting place" free text. Picking a thumbnail OR typing both resolve to one sub-scene `id` for scene 1. The B-010 matcher gets reused as the typed→sub-scene narrowing function.
- **Character picker: universal pool, theme-matching highlighted.** All 8 characters always visible regardless of theme. Each character carries `theme_fit: []`; characters fitting the picked theme get visual emphasis ("great match" badge + glow). Kid can still pick any. Required: ≥1 girl-coded + ≥1 boy-coded.
- **Theme picker required.** No "free-form / no theme" option. Cleaner UX; one codepath. The matcher stays alive in the codebase only as silent fallback for legacy pre-B-011 worlds.
- **Prompt preview reveal.** After all 5 inputs are filled, the form assembles the literal prompt as plain prose ("Create my realm in the Castle and Dragons world, starting at the drawbridge, starring my wizard. My goal is to find the dragon's lost egg. The twist is the dragon is shy.") and shows it to the kid. Educational beat: kids see how their ingredients become the prompt and build prompting intuition over time. Kid can hit Back to revise any step or "Summon Realm" to generate.
- **String ids + runtime validation, no literal unions.** Theme ids and sub-scene ids are `string` typed in the data model. A validator runs at module load and throws if any cross-references are broken (e.g., a sub-scene's `theme` doesn't match any theme, a `connects_to` points at a missing sub-scene, a `theme_fit` references a missing theme, a theme has 0 entry-candidates or 0 ending-candidates). Catches typos in dev immediately. Voice ids stay literal-typed (`"Fena" | "Ryan"`) since they bind to specific ElevenLabs IDs.
- **Backwards compat:** Old worlds (no `theme` column value) keep playing via the B-010 matcher path silently. No backfill, no migration of `worlds.map`.

## Architectural Decisions

### 1. New catalog files

`lib/themes-catalog.ts` (new):

```
Theme {
  id: string                    // "castle" | "forest" | etc, validated at module load
  label: string                 // "Castle and Dragons"
  description: string           // 1-line blurb for picker + Claude context
  thumbnail_path: string        // for theme picker UI
  sub_scenes: SubScene[]        // 15 entries
}

SubScene {
  id: string                    // namespaced: "castle_drawbridge"
  theme: string                 // matches a Theme.id
  label: string                 // "Drawbridge"
  description: string           // "wooden bridge over a moat" — feeds Claude narration
  file_path: string             // "/backgrounds/castle/drawbridge.svg"
  connects_to: string[]         // adjacent sub-scene ids
  can_be_entry: boolean
  can_be_ending: boolean
}
```

Export the 6 themes as a single array. Each theme contains its 15 sub-scenes inline.

`lib/characters-catalog.ts` (new):

```
Character {
  id: string                    // "knight"
  label: string                 // "Knight"
  description: string           // 1-line "armored, brave, swordfighter"
  thumbnail_path: string
  theme_fit: string[]           // theme ids this character fits well
  voice: "Fena" | "Ryan"
}
```

Export 8 characters. Required: ≥1 girl-coded + ≥1 boy-coded. Suggested list (CLI may adjust labels/fit):

1. Knight — boy-coded, voice Ryan, theme_fit: castle, forest
2. Wizard — girl-coded, voice Fena, theme_fit: castle, forest, space
3. Princess — girl-coded, voice Fena, theme_fit: castle, candy_land
4. Astronaut — boy-coded, voice Ryan, theme_fit: space, city
5. Mer-kid — neutral, voice Fena, theme_fit: underwater
6. Gingerbread Kid — neutral, voice Ryan, theme_fit: candy_land
7. Robot — neutral, voice Ryan, theme_fit: space, city
8. Dragon — neutral, voice Ryan, theme_fit: castle, forest

`lib/catalog-validator.ts` (new):
- Runs at module load (called from each catalog file's bottom).
- Checks: theme ids unique; sub-scene ids unique within theme + globally namespaced; every `theme` field references a real theme; every `connects_to` id references a real sub-scene IN THE SAME theme; every `theme_fit` references a real theme; every theme has ≥1 entry-candidate and ≥1 ending-candidate.
- Throws on init if any check fails. Caught in dev immediately.

**City:** subway entrance (entry), bus stop (entry), park, public library, school, skyscraper lobby, rooftop (ending), diner, comic shop, river bridge, museum (ending), alley, market square, movie theater (ending), underground tunnel (ending).

**Space:** spaceship bridge (entry), launch pad (entry), asteroid field, moon base, alien planet surface, black hole edge (ending), space station, crystal cave on Mars, nebula cloud, comet tail, star heart (ending), wormhole, robot factory, galaxy edge (ending), sun's outer ring (ending).

**Underwater:** sandy shore (entry), coral reef edge (entry), kelp forest, sunken ship, mermaid palace (ending), deep trench (ending), treasure cave, octopus garden, whale bones, underwater volcano, pearl cave (ending), reef cliffs, pirate wreck, lost city Atlantis (ending), abyss (ending).

CLI picks `connects_to` edges based on geographic intuition (e.g., drawbridge → outer gate → courtyard → great hall is a natural chain; tower stairs → tower top is direct; secret passage connects loosely across the castle). Each non-entry sub-scene should have ≥2 in-edges so the graph isn't a strict tree.

### 3. Placeholder SVG backgrounds (90 files)

`public/backgrounds/{theme}/{sub_scene_id}.svg` for each of the 90 entries. Each placeholder is ~5-10 lines of SVG:
- Same dimensions as existing backgrounds (audit current to match)
- Base color rect appropriate to theme (castle = stone gray, forest = green, candy = pink, city = blue-gray, space = dark navy, underwater = teal)
- 2-3 silhouette shapes hinting at the location (e.g., drawbridge = horizontal lines + chains; tower top = triangular peak + windows)
- Small label text in the bottom-right corner: "placeholder: castle_drawbridge" so it's obvious during dev which file is which

CLI agent generates each file inline via Claude during implementation. Don't try to make these look polished — they're stand-ins.

`public/characters/{character_id}.svg` for each of the 8 characters. Same approach.

`public/themes/{theme_id}.svg` for each of the 6 theme picker thumbnails. Same approach.

### 5. Landing form rewrite

`components/LandingForm.tsx`:

Step-by-step disclosure (not all at once). After each step is filled, the next step reveals.

**Step 1: Theme picker.** 6-card grid. Each card = thumbnail + label + 1-line description. Tap to select. Selected card shows checkmark. Reveals step 2.

**Step 2: Setting picker.** 15-card sub-scene grid filtered to picked theme. Each card = thumbnail + label. Tap to select. Below the grid: free text field "or describe a different starting place." If kid types, the existing B-010 matcher narrows to closest sub-scene id within this theme. Either path resolves to a single sub-scene id stored as `entry_sub_scene_id`. Reveals step 3.

**Step 3: Character picker.** 8-card grid. Theme-matching characters get a "great match" badge + subtle glow. Tap to select. Optional "name your hero" text field below. Reveals step 4.

**Step 4: Goal.** Single-line free text. Reveals step 5.

**Step 5: Twist.** Single-line free text. Reveals step 6.

**Step 6: Prompt preview.** Card displaying the assembled prompt as plain prose:

> "Create my realm in the **Castle and Dragons** world, starting at the **drawbridge**, starring my **wizard** named **Elara**. My goal is to **find the dragon's lost egg**. The twist is **the dragon is shy**."

Below the card: two buttons. **Back** (revises any step) and **Summon Realm** (generates).

State management: form holds `{ theme: string | null, entry_sub_scene_id: string | null, character_id: string | null, character_name: string, goal: string, twist: string }`. Reveal logic gates on the relevant fields being set.

Make sure the form is responsive — card grids reflow to 2 columns on narrow phones.

### 6. /api/generate route update

`app/api/generate/route.ts`:

- Request body shape: `{ theme, entry_sub_scene_id, character_id, character_name?, goal, twist }`. Old shape (free-text setting + free-text character) still accepted for legacy callers — coerced to no-theme + matcher path internally.
- For new theme-driven shape: validate theme exists in catalog; validate entry_sub_scene_id belongs to that theme and has `can_be_entry: true`; validate character_id exists; pass theme's full sub-scene list to `lib/claude.ts`'s generator as catalog context.
- Save `theme` to `worlds.theme` column on insert.

### 7. Claude prompt update

`lib/claude.ts`:

- Generator function gains parameters: `themeId: string, entrySubSceneId: string, themeSubScenes: SubScene[], characterId: string, characterName?: string`.
- Prompt structure adds a "library reference" block before the existing story instructions:
  > "Your story takes place in the [theme.label] world. Here are the available sub-scenes:
  > - `castle_drawbridge` (Drawbridge): wooden bridge over a moat. Connects to: outer_gate, courtyard. Entry: yes. Ending: no.
  > - ... (all 15 entries) ...
  >
  > Build your story tree using these sub-scenes. Each `scene.background_id` must be one of these ids. Scene 1 must be `[entrySubSceneId]`. Scenes 2-4 must follow library `connects_to` edges from the previous scene. Scenes 5+ may use any sub-scene from this theme. Ending scenes must use sub-scenes with Ending: yes. Bridge story-only when needed ('a hidden door appears...')."
- Hero voice resolution: read `voice` from the picked character in `lib/characters-catalog.ts`, pass to ElevenLabs. Replaces the B-010 hero_voice ad-hoc inference.

### 8. Parser validation extensions

`lib/claude.ts` parser:

- Every scene's `background_id` must reference a real sub-scene in the picked theme's catalog.
- Scene 1's `background_id` must equal `entry_sub_scene_id`.
- For scenes 2-4: `background_id` must appear in the previous scene's library `connects_to` array. (Adjacency constraint for the geographic half.)
- For scenes 5+: `background_id` must belong to the picked theme but no adjacency constraint.
- Ending scenes (`choices.length === 0` && `!is_choice_scene`): their `background_id` sub-scene must have `can_be_ending: true`.
- All existing B-009 flag + ending checks still apply.
- All existing B-010 phantom-requires + min-4-scene + level-2 pickup checks still apply.
- On any violation: throw, retry generation up to existing limits.

### 9. Renderer update

`components/StoryPlayer.tsx` / wherever scenes render:

- For new worlds: resolve `scene.background_id` via the catalog's `file_path`. Render the SVG.
- For old worlds (no theme): existing matcher / inline_svg path stays. No change.

### 10. Character voice plumbing

`lib/elevenlabs.ts`:
- `heroVoiceIdFor(name: "Fena" | "Ryan")` already exists from B-010. Keep.

`lib/claude.ts`:
- `tree.hero_voice` is now resolved from the picked character's `voice` field at generation time, not inferred by Claude. Remove the "Claude picks voice based on character vibe" instruction from the B-010 prompt; replace with "voice will be set server-side from character.voice."

## Smoke Tests

After CLI implements, Vanessa walks:

1. **Theme picker.** Landing shows 6 theme cards. Tap each in turn — only the picked card highlights. No other steps visible until a theme is picked.
2. **Setting picker reveals.** After theme = Castle, the 15 castle sub-scenes appear. Switch to Forest — the grid swaps to 15 forest sub-scenes. Setting field works either by tap or typing ("secret library" → matcher highlights/selects "library").
3. **Character picker.** All 8 characters always visible. With theme = Castle, knight + wizard + princess + dragon get a "great match" badge. Switch to Space — astronaut + robot get the badge instead. Picking any character works.
4. **Prompt preview.** After all 5 inputs filled, the assembled prompt card displays correctly with bolded ingredients. Back button revises any step. "Summon Realm" generates.
5. **Generated story scene 1.** Tree's scene 1 has `background_id` exactly equal to the kid's picked entry sub-scene id. Render shows that sub-scene's SVG.
6. **Adjacency for scenes 2-4.** Walk the tree. Each scene's `background_id` must appear in the previous scene's library `connects_to` array. Confirm across ~5 generations.
7. **Free-form scenes 5+.** Scenes 5+ can use any sub-scene from the theme regardless of adjacency.
8. **Ending sub-scenes.** All ending scenes' `background_id` is a sub-scene with `can_be_ending: true`. Walk all endings of ~3 realms.
9. **Catalog validator.** Manually break a `connects_to` reference in `lib/themes-catalog.ts` (e.g., point to a non-existent id). Run `npm run dev` — confirm the validator throws on init with a clear error message. Revert.
10. **Old worlds still play.** Open a pre-B-011 world from /profile (or use a saved one). Confirm it renders via the matcher fallback silently — no errors, no broken backgrounds.
11. **Hero voice.** Pick a girl-coded character (princess) in a Castle realm. Tap the hero in any scene during play. Voice = Fena. Pick a boy-coded character (knight). Voice = Ryan. Confirms B-010 voice routing still works through the catalog.
12. **Mobile responsive.** Resize browser to 375px wide. Theme grid reflows to 2 cols. Sub-scene grid reflows to 2 cols. Character grid reflows to 2 cols. Prompt preview wraps cleanly.
